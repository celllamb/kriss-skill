# HWPX Hancom Compatibility Notes

This note records HWPX behavior that was not clear from the public HWPX/OWPML materials, or that differed from the most natural interpretation of the public XML structure. Use it when generating HWPX reports with embedded images, especially when the file must open reliably in desktop Hancom HWP.

## Scope

These findings come from KRISS HWPX trip-report repair work involving image-heavy evidence and presentation figures. They are based on:

- Direct HWPX ZIP/XML inspection.
- Desktop Hancom HWP COM automation with `HWPFrame.HwpObject`.
- `InsertPicture` sample files generated by HWP itself.
- HWP open/save round-trip comparison.
- HWP PDF export followed by page-image/contact-sheet inspection.

Treat the items below as Hancom Office compatibility rules, not as a replacement for the official format documents.

## Official Documents Used As Baseline

- Hancom publishes HWP/HWPML/OWPML format material and notes that OWPML is the open HWP document format, with `.hwpx` support in older compatible versions and `.owpml` support from HWP 2018 onward: https://www.hancom.com/support/downloadCenter/hwpOwpml
- Hancom Help documents picture insertion as an embedded or linked picture workflow, with options such as in-line placement, size adjustment, and fit-to-cell behavior: https://help.hancom.com/hoffice100/en-US/Hwp/insert/figure/figure.htm
- Hancom Help documents object size and position as fixed or relative values, with wrapping/placement options such as in-line, square, and top-and-bottom: https://help.hancom.com/hoffice/multi/en_us/hwp/insert/objectattribute/objectattribute%28general%29.htm
- Hancom Developer exposes `HwpCtrl.InsertPicture`, but the public page does not fully explain the HWPX XML that desktop HWP writes after insertion: https://developer.hancom.com/en-us/webhwp/devguide/hwpctrl/methods/insertpicture

## Main Compatibility Findings

### 1. HWP-compatible picture XML differs from the simplest spec-shaped XML

The failing generated reports used valid package entries and decodable image files, but some pictures rendered as grey/black placeholders or caused unstable HWP open behavior. The fix was not merely "valid XML"; the generated `hp:pic` object needed to mimic a picture object produced by HWP's own `InsertPicture`.

For simple embedded pictures, the HWP-generated pattern is:

```xml
<hp:pic textWrap="TOP_AND_BOTTOM" textFlow="BOTH_SIDES" numberingType="PICTURE">
  <hp:offset x="0" y="0"/>
  <hp:orgSz width="{target_display_hwp_unit}" height="{target_display_hwp_unit}"/>
  <hp:curSz width="0" height="0"/>
  <hp:flip horizontal="0" vertical="0"/>
  <hp:rotationInfo angle="0" centerX="{target_w/2}" centerY="{target_h/2}" rotateimage="1"/>
  <hp:renderingInfo>
    <hc:transMatrix e1="1" e2="0" e3="0" e4="0" e5="1" e6="0"/>
    <hc:scaMatrix e1="1" e2="0" e3="0" e4="0" e5="1" e6="0"/>
    <hc:rotMatrix e1="1" e2="0" e3="0" e4="0" e5="1" e6="0"/>
  </hp:renderingInfo>
  <hp:imgRect>
    <hc:pt0 x="0" y="0"/>
    <hc:pt1 x="{target_w}" y="0"/>
    <hc:pt2 x="{target_w}" y="{target_h}"/>
    <hc:pt3 x="0" y="{target_h}"/>
  </hp:imgRect>
  <hp:imgClip left="0" right="{source_image_unit_w}" top="0" bottom="{source_image_unit_h}"/>
  <hp:inMargin left="0" right="0" top="0" bottom="0"/>
  <hp:imgDim dimwidth="{source_image_unit_w}" dimheight="{source_image_unit_h}"/>
  <hc:img binaryItemIDRef="imageN" bright="0" contrast="0" effect="REAL_PIC" alpha="0"/>
  <hp:effects/>
  <hp:sz width="{target_w}" widthRelTo="ABSOLUTE" height="{target_h}" heightRelTo="ABSOLUTE" protect="0"/>
  <hp:pos treatAsChar="1" affectLSpacing="0" flowWithText="1" allowOverlap="0"
          holdAnchorAndSO="0" vertRelTo="PARA" horzRelTo="COLUMN"
          vertAlign="TOP" horzAlign="LEFT" vertOffset="0" horzOffset="0"/>
  <hp:outMargin left="0" right="0" top="0" bottom="0"/>
</hp:pic>
```

Practical rules:

- Use `curSz width="0" height="0"`, even though `orgSz` and `sz` hold the visible display size.
- Use `rotateimage="1"`.
- Keep `transMatrix`, `scaMatrix`, and `rotMatrix` as identity matrices.
- Add a sibling empty `hp:t` after `hp:pic` inside the same `hp:run`; HWP-generated picture runs include it.
- Remove stale `hp:linesegarray` layout cache from copied paragraphs and let HWP recompute layout.

### 2. Display units and source-image units are not the same

Do not use source pixel dimensions as visible HWP picture dimensions.

Observed display-size conversion:

```text
target_hwp_unit = round(target_mm / 25.4 * 7200)
```

Example from desktop HWP `InsertPicture`:

- Inserting a 2000 x 1125 PNG with target width/height arguments `150, 84` produced `orgSz width="42520" height="23811"`.
- `150 / 25.4 * 7200 = 42519.7`, so desktop HWP is effectively writing 7200 HWP display units per inch.

Important failure mode:

- Passing `36000` as an `InsertPicture` width because it "looks like a HWP unit" makes a huge picture. It may look blank or cropped after PDF export because the image is far outside the page.

Source image crop/extent units:

- `imgClip` and `imgDim` are original-image coordinate units, not display HWP units.
- HWP's output was approximately proportional to pixels, around `pixel * 75`, but exact values can vary slightly with image metadata and rounding.
- Keep `imgClip` and `imgDim` internally consistent with each other; do not force them to equal `orgSz`/`sz`.

### 3. Image payload format matters less by spec than by HWP renderer behavior

HWP Help lists many supported picture types, including PNG, JPG, and BMP. In generated HWPX packages, the most stable result in this case came from mimicking HWP's own output:

- Store embedded slide/evidence images as RGB PNG under `BinData/`.
- Add a matching `Contents/content.hpf` manifest item, e.g. `id="image1" href="BinData/image1.png" media-type="image/png" isEmbeded="1"`.
- Store `BinData/` image ZIP entries uncompressed (`ZIP_STORED`).
- Cap very large images before embedding. In this skill, `MAX_EMBEDDED_PIXEL_EDGE = 2200` was sufficient for readability and stability.

Observed failed or risky approaches:

- Hand-authored JPEG picture objects with spec-looking dimensions opened but still produced grey/black placeholders in HWP PDF export.
- BMP conversion made output much larger and did not solve the rendering issue.
- Progressive JPEGs and double-compressed image entries should be avoided for compatibility.

### 4. Valid HWPX is not enough; HWP render/export must be tested

The broken reports passed basic ZIP/XML and image-decode checks. The failure appeared only when desktop HWP laid out or exported the document.

Minimum checks for image-bearing reports:

1. Run general HWPX validation.
2. Run image-reference validation:
   - every `hc:img binaryItemIDRef` has a manifest item;
   - every manifest image exists in `BinData/`;
   - every image decodes;
   - media type and extension match;
   - `hp:pic` sizes are plausible.
3. Open the file in desktop HWP.
4. Save as a separate HWPX file, never over the generated original.
5. Compare original vs HWP-saved package.
6. Export to PDF from HWP.
7. Render the PDF to page images and inspect a contact sheet.

The contact sheet matters because HWP can open a file successfully while still rendering individual picture controls as placeholders.

### 5. HWP round-trip changes are expected, but image payload changes are not

Desktop HWP rewrites parts of a generated package when saving. This does not necessarily mean the generated file is bad.

In the repaired report, HWP changed only:

- `Contents/content.hpf`
- `Contents/header.xml`
- `Contents/section0.xml`
- `Preview/PrvImage.png`
- `Preview/PrvText.txt`
- container/version metadata such as `META-INF/container.rdf`, `META-INF/container.xml`, and `version.xml`

The important pass condition was:

```text
image_payloads_unchanged = true
only_hancom_normalized_entries = true
```

Accept HWP's round-trip when:

- no package entries are added or removed;
- all `BinData/` image hashes are unchanged;
- only HWP-normalized XML/preview/metadata entries differ.

Reject or revise when:

- a `BinData/` image is removed, renamed without matching manifest repair, or re-encoded;
- `hc:img` references are lost;
- HWP produces a file that opens only after repeated crash/restart behavior;
- PDF export still shows black/grey placeholders or severe cropping.

### 6. The generated original may remain unchanged even when HWP repairs a separate save

For this case, opening the generated HWPX did not silently modify the original file on disk. The original file's hash and timestamp remained stable. HWP's changes appeared only in the explicitly saved round-trip copy.

Still, do not assume this will always hold. For future projects:

- record the generated file hash immediately after creation;
- open it in HWP;
- check the generated file hash and timestamp again;
- save HWP's version to a separate path;
- compare the two packages.

### 7. Current skill implementation rules

For `kriss-trip-report`, keep these generation rules unless a new HWP version proves otherwise:

- Use PNG `BinData/imageN.png` payloads, not hand-authored JPEG/BMP variants.
- Keep `BinData/` ZIP entries uncompressed.
- Use HWP-compatible `hp:pic` fields:
  - `orgSz`/`imgRect`/`sz` in display HWP units;
  - `curSz = 0`;
  - `imgClip`/`imgDim` in source-image units;
  - identity rendering matrices.
- Use target display width around 150 mm for normal report-column images.
- Use full-page evidence images only after privacy masking.
- Validate, HWP round-trip, compare, PDF export, and contact-sheet inspect before delivery.

## Final Rule Of Thumb

The official HWPX/OWPML structure is necessary but not sufficient for desktop HWP compatibility. For picture-heavy generated reports, trust HWP's own `InsertPicture` output shape, then verify with HWP round-trip and PDF rendering.
