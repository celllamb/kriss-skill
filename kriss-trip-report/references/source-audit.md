# 원자료 대조 감사

KRISS 국외출장보고서를 최종화하기 전, 또는 기존 보고서를 원자료와 대조할 때 이 문서를 사용한다.

## 독립 컨텍스트 감사

subagent를 사용할 수 있으면, 실질적 회의형/학회형/파견연구형 보고서 최종 전달 전 독립 감사를 수행한다.

- 현재 컨텍스트를 fork하지 않은 새 subagent를 시작한다.
- 원자료만 전달한다. 예: 초안 보고서 경로 또는 추출 텍스트, 원자료 폴더/파일, 출장 유형, 감사 체크리스트.
- 사용자가 표적 검증을 명시하지 않는 한 의심되는 오류, 수정 의도, 요약, 기존 결론을 전달하지 않는다.
- 감사 결과는 다음 범주로 묶게 한다: 자동 수정 가능한 확정 불일치, 사용자 확인 필요, 약하거나 과도하게 특정적인 주장, 원자료 간 충돌, 연구번호/내용 불일치, 레이아웃/렌더링 위험, 슬라이드 첨부 적합성 문제, HWPX package 호환성 문제, 증빙 누락, 개인정보/증빙 문제.
- 감사자는 공동 작성자가 아니라 원자료 비판 검토자로 취급한다.

## 감사 에이전트 구성

일반 보고서는 독립 일반 감사자 1명을 기본으로 한다. 원자료가 많거나 기술적으로 조밀하거나 이미 불일치 징후가 있으면 복수의 전문 감사자를 사용한다.

감사자 1명이 적합한 경우:

- 실질 원자료가 대략 10개 미만
- 출장 분야가 하나의 회의/학회 영역이고 연구번호가 적음
- 주요 위험이 단순 추출 정확도, 날짜, 성명, 증빙 배치 정도임
- 최대 recall보다 처리 속도가 중요함

복수 전문 감사자가 적합한 경우:

- `K`와 `P` 연구가 섞인 비교연구/pilot 식별자가 많음
- 원자료가 여행 증빙, 아젠다/action, 기술 비교결과, CMC claim, 개인정보/증빙 이미지 등 여러 영역에 걸침
- 이전 감사에서 연구번호 drift, 원자료 충돌, 사용자 확인 항목이 발견됨
- 최종 보고서가 공식 기록으로 재사용되어 누락된 모순의 비용이 큼

권장 전문 감사자 분리:

- `기술 연구 감사자`: 연구번호, 분석대상, 측정대상, 물질, 참여기관, 결과, CMC/HFTLS 문구, 약한 기술 주장
- `일정/action 감사자`: 회의 날짜, 아젠다 순서, 변경 일정, action 담당자, 마감일, wrap-up 충돌
- `증빙/개인정보 감사자`: 출장기간, 항공권/출입국 증빙, 수집자료 배치, 이미지 내 보호 식별자
- `레이아웃/첨부 감사자`: 긴 bullet, 혼합 한국어/영어 줄바꿈 위험, 이미지 배치, 슬라이드 관련성, 한 문장 이미지 근거
- `HWPX package 감사자`: HWP2018 열기 가능성, `BinData` payload 변경, `content.hpf` manifest 변경, 이미지 참조 무결성, Hancom round-trip 비교

여러 감사자가 반환하면 메인 에이전트가 결과를 조정한다. 전문 감사자가 보고서를 직접 수정하게 하거나 투표로 충돌을 없애지 않는다. 감사자 의견이 갈리면 충돌을 보존하고, 해결이 회의 맥락이나 구두 확인에 달려 있으면 사용자에게 묻는다.

## 감사 체크리스트

가장 우선순위 높은 원자료와 대조해 다음을 확인한다.

- 출장기간, 출국/입국, 도시, 국가, 방문기관
- 회의명, 회의일, 장소, 세션 제목, 일자별 아젠다 순서
- 원자료-아젠다 매칭: 실질적 발표자료나 기술 원자료가 보고서 본문에 쓰이기 전에 공식 아젠다/프로그램 항목, action/회의록 항목, 증빙 범주, 사용자 확인 맥락 중 하나에 연결되었는지
- 원자료 날짜와 행사 출처: 특정 날짜 회의의 슬라이드가 다른 날짜 또는 합동회의 원자료처럼 취급되지 않았는지. 나중 회의일에 아젠다/메모/회의록/wrap-up만 있으면 작업 메모와 추적표에 그렇게 분류하고, 최종 본문에는 작성 근거 설명을 쓰지 않았는지
- 출장자와 면담자 이름, 역할, 기관, 국가, 면담일
- 비교연구/pilot 참여, 발표 결과, 후속 action, 담당자, 마감일
- 연구번호와 연구 내용: 모든 `K`, `P`, pilot, comparison, task group 번호가 올바른 분석대상, 측정대상, 물질, 참여기관, 결과 유형, 후속 action과 연결되는지
- KRISS 기여, 참여, task group/team 참여 예정, 향후 action에 대한 주장
- 수집자료와 여행 증빙의 배치
- 첨부 전 증빙 이미지의 보호 식별자 노출 여부
- 렌더링 및 가독성 위험: 긴 bullet, 조밀한 혼합언어 기술 문구, 좁은 표 셀, 긴 무공백 token, HWPX/Hancom에서 겹치거나 잘릴 수 있는 문구
- 삽입된 슬라이드/이미지: 모든 비증빙 이미지는 주변 아젠다를 뒷받침하고, 텍스트로 대체하기 어려운 시각 정보를 가지며, 한 문장으로 명확한 관련성을 설명할 수 있어야 함
- HWPX package 호환성: XML 검증 통과만으로 충분하지 않다. HWP2018이 손상/변조 경고를 내는지, image payload, `BinData`, `Contents/content.hpf`가 수작업 추가/삭제/재압축/불일치되었는지 확인한다. bullet 단축 후 실제 문단 길이를 넘는 `hp:lineseg textpos`도 확인한다.
- 독자 이해도: 회의 내부 약어와 영어 표현은 회의 미참석 측정표준 종사자도 이해할 수 있어야 한다. `HFTLS`, `CMC claim`, `measurand`, `anchor KC`, `bridge pilot` 같은 용어는 한국어 설명을 먼저 또는 함께 쓴다.
- 본문 출처 설명 노출: 최종 보고서 bullet은 회의 내용만 말해야 한다. `근거`, `회의 메모`, `발표자료 없음`, `자료상`, `아젠다 기반`, `메모 기반`, 자료 분류 설명이 본문에 있으면 감사 finding으로 잡는다.
- 기관 비중 균형: 출장의 주요 가치가 타 NMI 활동, 방법, 비교연구 진행상황, action item, 국제 동향 확인인 경우 KRISS 발표 세부 결과가 과도하게 강조되지 않았는지 확인한다.

## 감사 결과 조정 규칙

- 확정 불일치는 전달 전 수정한다.
- 원자료 위계가 명확하고 외부 맥락이 필요 없을 때만 자동 수정한다. 예: 잘못된 분석대상/유전자, 잘못된 참여기관 수, source가 no participants라고 하는데 참여를 암시한 문장, source 제목과 결과 슬라이드가 명확히 분리하는 연구번호 혼용.
- 맥락 의존 충돌은 자동 수정하지 않는다. 다음은 `사용자 확인 필요` 목록에 넣는다.
  - 회의 중 또는 발표 아젠다 이후 일정이 변경됨
  - 나중 wrap-up이 이전 슬라이드나 아젠다와 충돌함
  - 선물 여부, 직접 면담, 구두합의처럼 제공 파일에 없는 출장자 확인 사실을 보고서가 사용함
  - 첨부되지 않은 기관 맥락, 이메일, 구두 합의에 의존함
- 약한 주장은 원자료가 지지하는 수준으로 낮춘다. 참여나 확정 commitment가 명시되지 않으면 `논의`, `동향 파악`, `모니터링`, `후속 검토 필요` 등을 사용한다.
- 공식 세션 bundle 명칭은 유지할 수 있지만, 세션 제목만으로 참여를 추론하지 않는다.
- 원자료가 병렬 pilot 또는 연구에 참여자가 없었다고 하면 해당 pilot/연구 참여를 암시하는 표현을 피한다.
- 원자료가 충돌하면 `source-materials.md`의 우선순위를 따른다. 그래도 해결되지 않으면 충돌을 표시하고 강한 주장 전 사용자에게 묻는다.
- 원자료가 아젠다/프로그램 항목 또는 명확한 행정/증빙 역할에 연결되지 않으면 사용하지 않는다. confirmation queue에 넣고 사용자가 의도한 회의 맥락을 묻는다.
- 슬라이드나 그림이 아젠다 민감 자료이면 source-to-agenda map으로 배치를 확인한다. Good News Story 그림을 같은 주제가 나중 workshop deck에 보인다는 이유만으로 나중 workshop 아래에 두지 않는다.
- 수정 후 HWPX를 다시 추출하고 검증한다.

## 레이아웃 및 슬라이드 첨부 감사

레이아웃 문제는 단순 꾸밈 문제가 아니라 감사 finding이다. 기술적으로 맞아도 렌더링이 나쁠 문구는 잡아 수정한다.

- study ID, 화학/물질명, `copy number concentration`, `extraction efficiency`, ratio, 날짜, 참여기관 목록 등 영어 기술구가 많이 섞인 긴 bullet을 flag한다.
- 시각적으로 불안정한 긴 한 문장보다 짧은 bullet을 선호한다. 여러 사실을 담아야 하면 인접 bullet로 나누거나 정확도를 해치지 않는 범위에서 영어 표현을 한국어로 압축한다.
- 의미를 유지할 수 있으면 slash가 많은 표현이나 긴 무공백 token을 피한다.
- HWPX 생성 후 텍스트를 다시 추출해 위험 bullet을 스캔한다. Hancom 렌더러 또는 시각 렌더러가 있으면 긴 혼합언어 bullet과 삽입 이미지 페이지를 직접 확인한다.

첨부 슬라이드 규칙:

- 아젠다당 대표 이미지는 원칙적으로 1개를 권장하되, 시각적 가치 테스트를 통과할 때만 사용한다.
- 제거된 슬라이드가 중요하지만 텍스트 중심이면 같은 아젠다 원자료에서 더 나은 그래프, 모델, workflow, 개념도가 있는지 먼저 찾는다.
- 모든 비증빙 슬라이드는 다음 테스트를 통과해야 한다: “이 슬라이드는 ___를 시각적으로 보여 주며, 보고서의 ___ 문장을 뒷받침하고, 텍스트로 줄이면 ___가 손실된다.”
- 결과 패턴, 그래프, 실험 workflow, 측정모델, 개념도, 수식/공식 배열, 의미 있는 시각 비교가 있을 때만 유지한다.
- 일정, action 목록, 결과 요약표, 로드맵 텍스트, 제목 슬라이드, 참석자 목록, 장소/등록 안내, 일반 텍스트 bullet처럼 명확히 본문으로 요약 가능한 슬라이드는 제거하거나 대체한다.
- 제거된 슬라이드에 중요한 정보가 있으면 보고서 본문 또는 source trace matrix에 반영해 정보가 사라지지 않게 한다.
- 이미지 관련성 설명은 작업 메모, trace matrix, image `shapeComment`, 주변 보고서 note 중 하나에 남겨 나중 감사자가 확인할 수 있게 한다.
- 항공권, 보딩패스, 출입국 사실증명 같은 증빙 이미지는 기술 슬라이드 관련성 테스트 대상이 아니지만 개인정보와 가독성 검토는 필요하다.

## HWPX package 호환성 감사

HWPX 구조 검증기가 XML과 참조 오류를 잡아도, Hancom Office 2018은 수작업 repack된 package를 의심해 거부하거나 경고할 수 있다.

- HWP2018 전달본에서는 Hancom Office에서 다시 열고 저장해 성공하지 않은 한 `BinData/*`, `Contents/content.hpf` manifest, image payload를 수작업 추가/삭제하지 않는다.
- 기존 이미지 포함 보고서에서 부적합한 슬라이드를 제거할 때는 package payload와 manifest를 보존하고 보이는 `hp:pic` 문단만 제거하는 편이 안전하다. 미참조 `BinData` 경고가 남을 수 있지만 HWP2018에서는 package membership 변경보다 안전한 경우가 많다.
- 최종 파일에는 HWPX XML 검증과 `scripts/validate_hwpx_images.py`를 실행한다. 이전 HWP2018 열기 가능 base가 있으면 `scripts/compare_hwpx_packages.py base.hwpx final.hwpx`도 실행한다.
- Hancom Office가 설치되어 있으면 Hancom engine으로 실제 openability를 확인한다. `Open=False`이거나 손상/보안 경고가 뜨면 ZIP/XML/image 검증이 통과해도 deliverable이 아니다.
- `image_payloads_unchanged`가 true이고 payload 추가/삭제가 없으며 변경 항목이 Hancom-normalized XML 또는 수정 section으로 제한되면 더 안전한 최종본으로 본다. image payload가 변경/삭제되거나 새 manifest 항목이 생기면 전달 위험으로 본다.
- 대체 이미지가 필요하지만 수작업 package 변경이 요구되면, 이미지를 보류하거나 Hancom Office round-trip 삽입을 사용하거나 HWP2018 open/save 검증이 필요한 draft로 명확히 표시한다.
- 보이는 슬라이드만 제거하고 미참조 `BinData`를 의도적으로 남기는 등 호환성 tradeoff는 작업 메모에 기록한다.

## Source Trace Matrix

사용자가 수정 보고서가 개별 원자료와 일치하는지 묻는 경우, 서술형 메모만 쓰지 말고 간결한 trace matrix를 만든다.

포함 열:

- `Report claim`: 보고서 문장 또는 bullet을 가능한 짧게 요약
- `Source`: 파일명과 가능한 경우 page/slide/sheet/row
- `Agenda item`: 공식 아젠다 날짜/항목/세션, 또는 기술 아젠다가 아닌 경우 증빙/행정 범주
- `Status`: `matched`, `corrected`, `needs user confirmation`, `user-confirmed context`, `source conflict`, `not supported`
- `Resolution`: 최종 보고서 문구 또는 다음 action

고위험 항목은 모두 추적한다. 예: 출장 날짜, 회의 날짜, 연구번호, 연구 내용, 참여/결과 주장, action 마감일, CMC/HFTLS 문구, task group 참여, 증빙 배치. 큰 보고서는 저위험 행정 claim은 대표적으로만 추적하고 고위험 기술 claim은 모두 추적한다.

## 사용자 확인 맥락

일부 공식 보고 사실은 제공된 파일이 아니라 출장자 확인에서 온다. 이를 unsupported claim과 구분한다.

- 사용자가 구두합의, 초청, 참여 의사 표명, 회의 중 일정 변경, 선물 상태, 제공 파일에 없는 직접 논의를 확인하면 `user-confirmed context`로 분류한다.
- 관련성이 있으면 보고서에 보존하되 확인 수준에 맞춰 표현한다. 예: 사용자가 formal acceptance를 확인하지 않는 한 `참여 확정`이 아니라 `참여 의사 표명`으로 구분한다.
- 나중 감사자가 unsupported로 제거하지 않도록 audit memo 또는 trace matrix에 사용자 확인 사실을 기록한다.
- 원자료가 직접 충돌하지 않는 한 다른 감사자에게 사용자 확인 맥락을 추론으로 부정하게 하지 않는다.

## 연구번호 점검

기술회의 보고서에서 study-number drift는 고위험 오류이다. 감사에서 명시적으로 확인한다.

- 원자료에서 compact mapping table을 만든다. 열 예: 연구번호, 제목, 분석대상/물질, 측정대상, coordinator/presenter, 참여기관, 상태, 다음 action, source file.
- mapping table과 초안 보고서를 비교한다. 연구번호가 다른 연구의 유전자, 물질, 결과, 참여기관, 마감일, action과 짝지어진 줄을 flag한다.
- 세션 bundle을 조심한다. `K190/P249/P250` 세션 제목은 맞을 수 있지만 실제 결과는 `K190`과 `P250`에만 적용될 수 있다.
- 후속 action은 연구번호별로 분리한다. `K198` action을 `K189` 아래에 두거나, `P250` 절대농도 결과를 `P249` 결과처럼 쓰지 않는다.
- 연구번호가 제목에만 나오고 실제 참여나 결과에는 나오지 않으면 이를 명시적으로 구분한다.

## 사용자 확인 목록

원자료 충돌 해결을 사용자에게 물을 때는 필요한 맥락만 간단히 제공한다.

- 충돌하는 원자료와 정확한 날짜/용어를 말한다.
- 가능한 해석을 설명하되 사실로 단정하지 않는다.
- 직접적인 질문 하나와 제안 보고서 문구를 함께 제시한다.

예:

`NAWG-26-036 제목에는 November 2026이 있지만 프로그램에는 19-20 Oct / 21 Nov / 22 Nov가 함께 보입니다. 한국어 wrap-up과 사용자 설명은 APMP 일정 중복 때문에 회의 중 2026.10.19.-10.22.로 조정되었다는 맥락입니다. 보고서에는 확정 조정 일정으로 쓸까요, 아니면 조정안 확인 필요로 쓸까요?`
